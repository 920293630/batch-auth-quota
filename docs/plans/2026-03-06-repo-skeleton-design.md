# batch-auth-quota 仓库骨架设计

## 目标

将 `batch-auth-quota` 从“可直接运行的单目录脚本”整理为“可单独维护、可打包安装、可初始化发布”的轻量独立仓库，同时不改变现有批量额度检查能力。

## 设计原则

- 保持轻量：仍以单文件主脚本为核心，不额外拆复杂包结构
- 保持兼容：继续支持 `python3 batch_auth_quota.py` 与 `bash run.sh`
- 便于分发：增加 `pyproject.toml`，支持 `pip install -e .`
- 便于维护：补齐版本号、变更记录、发布清单与忽略规则
- 避免误导：在未确认开源协议前，不擅自写入正式 `LICENSE`

## 方案对比

### 方案 A：最小仓库骨架（推荐）

保留根目录脚本 `batch_auth_quota.py`，新增 `pyproject.toml`、`VERSION`、`CHANGELOG.md`、`RELEASE.md`、`docs/plans/`，并初始化独立 `git` 仓库。

优点：
- 改动小
- 学习成本低
- 不影响现有使用方式
- 后续可以平滑演进为正式 Python 包

缺点：
- 代码仍为单文件，长期演进时模块化略弱

### 方案 B：立即重构为包结构

将脚本拆成 `src/batch_auth_quota/` 包，命令入口、统计逻辑、配置逻辑分模块。

优点：
- 更标准
- 长期维护更清晰

缺点：
- 改动大
- 容易影响当前脚本稳定性
- 超出本轮“独立抽离工具”最小目标

### 方案 C：仅初始化 git，不补打包元数据

优点：
- 最快

缺点：
- 仍不方便安装、发布、版本管理
- 仓库骨架价值有限

## 结论

采用方案 A：

1. 新增 Python 项目元数据与控制台入口
2. 增加版本与发布文档
3. 初始化独立 `git` 仓库
4. 保持原有脚本运行方式不变

## 范围

本轮包含：
- 仓库元数据
- 发布文档
- 版本标识
- git 初始化
- 基础本地校验

本轮不包含：
- 大规模代码重构
- 正式 license 选型
- 远程仓库推送
- 自动化 CI 发布流程
